# 퀴즈 유형 및 패턴

## 개요

학습 섹션 완료 후 이해도를 확인하는 퀴즈를 생성한다.
난이도, 유형, 힌트 전략을 체계적으로 관리한다.

---

## 퀴즈 유형

### ⭐ 개념 확인 (Concept Check)

가장 기본적인 이해도를 확인하는 문제.

#### 유형 A: 정의 선택
```
다음 중 [개념]에 대한 올바른 설명은?

A) [오답 - 흔한 오해]
B) [정답]
C) [오답 - 유사 개념과의 혼동]
D) [오답 - 부분적으로 맞지만 불완전]
```

#### 유형 B: 참/거짓 (T/F)
```
다음 설명이 맞으면 O, 틀리면 X를 선택하세요.

1. [진술 1] → O / X
2. [진술 2] → O / X
3. [진술 3] → O / X
```

#### 유형 C: 빈칸 채우기
```
[개념]은 ___를 위해 사용되며, ___와 함께 사용할 수 있습니다.
```

### ⭐⭐ 코드 분석 (Code Analysis)

코드를 읽고 동작을 예측하는 문제.

#### 유형 A: 출력 예측
```
다음 코드의 출력 결과는 무엇일까요?

[코드 블록]

A) [출력 A]
B) [출력 B]
C) [출력 C]
D) 에러 발생
```

#### 유형 B: 버그 찾기
```
다음 코드에는 버그가 있습니다. 문제점과 해결 방법을 설명하세요.

[버그가 있는 코드 블록]
```

#### 유형 C: 코드 비교
```
다음 두 코드의 차이점을 설명하세요.

// 코드 A
[코드 블록 A]

// 코드 B
[코드 블록 B]
```

### ⭐⭐⭐ 실습 문제 (Hands-on)

직접 코드를 작성하는 문제.

#### 유형 A: 기능 구현
```
다음 요구사항을 만족하는 코드를 작성하세요.

요구사항:
- [요구사항 1]
- [요구사항 2]
- [요구사항 3]

시작 코드:
[부분적으로 작성된 코드]
```

#### 유형 B: 최적화
```
다음 코드는 동작하지만 [문제점]이 있습니다.
성능을 개선한 버전을 작성하세요.

[최적화 대상 코드]
```

#### 유형 C: 리팩토링
```
다음 코드를 [패턴/원칙]에 맞게 리팩토링하세요.

[리팩토링 대상 코드]
```

## 난이도 배분

### 학습 단계별 퀴즈 난이도 비율

| 학습 단계 | ⭐ 개념 확인 | ⭐⭐ 코드 분석 | ⭐⭐⭐ 실습 |
|----------|-------------|--------------|----------|
| 🟢 기초 | 60% (3문제) | 40% (2문제) | 0% |
| 🟡 중급 | 20% (1문제) | 60% (3문제) | 20% (1문제) |
| 🔴 고급 | 0% | 40% (2문제) | 60% (3문제) |

### 한 세트 구성

- 기본: **5문제** 한 세트
- 사용자 요청 시: 3문제 또는 7문제로 조절 가능
- 문제 순서: 쉬운 것 → 어려운 것 순서로 배치

## 힌트 전략

### 난이도별 힌트 수준

| 난이도 | 힌트 유형 | 예시 |
|--------|----------|------|
| ⭐ | **직접적 힌트** - 정답과 가까운 단서 | "이 개념은 '상태'와 관련이 있습니다." |
| ⭐⭐ | **방향성 힌트** - 어떤 방향으로 생각할지 | "코드가 실행되는 순서를 따라가 보세요." |
| ⭐⭐⭐ | **구조 힌트** - 접근 방법의 뼈대만 | "먼저 어떤 데이터가 필요한지 정리해보세요." |

### 힌트 제공 규칙

1. 사용자가 "힌트" 요청 시에만 제공 (자동 제공하지 않음)
2. 한 문제당 최대 2번의 힌트 제공
3. 두 번째 힌트는 첫 번째보다 더 구체적으로 제공
4. 힌트 제공 후에도 모르겠으면 해설로 전환

## 피드백 패턴

### 정답 피드백

```
✅ 정답입니다!
[왜 이것이 정답인지 한 줄 설명]
```

추가로 "이 개념을 더 확장하면..." 같은 심화 정보를 짧게 제공할 수 있다.

### 오답 피드백

```
❌ 아쉽지만 틀렸습니다.

정답: [정답]
설명: [왜 이것이 정답이고, 사용자의 답이 왜 틀렸는지]

💡 팁: [관련 개념을 다시 이해하기 위한 한 줄 조언]
```

### 부분 정답 피드백 (실습 문제)

```
🔶 거의 맞았습니다!

좋은 점: [잘한 부분]
개선할 점: [부족한 부분]

수정 제안:
[개선된 코드 또는 설명]
```

## 퀴즈 결과 요약

### 결과 형식

```
📊 퀴즈 결과: [정답 수]/[전체 수] 정답 ([퍼센트]%)

✅ 문제 [번호들] - 정답
❌ 문제 [번호] - [약한 부분 요약]
❌ 문제 [번호] - [약한 부분 요약]

💡 추천: [약한 부분에 대한 복습 가이드]
```

### 점수별 피드백

| 점수 | 피드백 |
|------|--------|
| 100% | "완벽합니다! 이 섹션을 확실히 이해하셨네요. 다음으로 넘어가볼까요?" |
| 60~80% | "대부분 잘 이해하셨습니다! 틀린 부분만 간단히 복습하고 넘어가면 좋겠습니다." |
| 40~60% | "핵심 개념을 다시 정리하면 도움이 될 것 같습니다. 복습 후 다시 시도해볼까요?" |
| 0~40% | "이 부분은 조금 더 시간을 들여 학습하면 좋겠습니다. 다시 설명해드릴까요?" |

## 퀴즈 생성 시 주의사항

1. **원문서 기반**: 학습한 섹션의 내용에서만 출제 (범위 벗어나지 않기)
2. **명확한 정답**: 모호하거나 논쟁의 여지가 있는 문제는 피함
3. **공정한 난이도**: 트릭 문제나 함정 문제는 지양
4. **실용성**: 실제 개발에서 마주칠 수 있는 상황 기반
5. **코드 예시**: 문법적으로 올바른 코드만 사용 (의도적 버그 찾기 문제 제외)
6. **한국어 주석**: 퀴즈의 코드 블록에도 한국어 주석 포함
